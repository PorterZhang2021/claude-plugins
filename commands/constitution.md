# Constitution Workshop

帮助用户为新项目结对编写开发宪法（constitution.md）。

## 目标

引导用户逐条确定项目的核心开发原则，最终生成 `.claude/constitution.md` 文件。

## 流程

### Step 1：了解项目背景

询问用户：
- 这是什么类型的项目（Web 服务、CLI 工具、数据管道等）？
- 团队规模和技术栈？
- 有没有特别想强调或想避免的开发习惯？

### Step 2：逐条共建原则

基于项目背景，依次引导用户讨论以下方向（每条先给出草稿，用户确认或修改）：

1. **简单性原则** — 克制、直白优于聪明、单一职责、显式优于隐式、边界
2. **测试原则** — 是否测试先行？单元/集成/端到端的边界如何划定？实现前是否强制拆分为可独立验证的子任务（每个子任务对应至少一个测试用例）？
3. **明确性原则** — 错误处理策略、命名规范、禁止的模式（如全局变量）
4. **（可选）其他原则** — 根据项目特点补充，如安全性、性能、可观测性等

每条原则讨论完后立即写入文件，不等全部完成再写。

### Step 3：生成文件

在项目根目录创建 `.claude/constitution.md`，结构如下：

```
# 开发宪法

## 第N条：[原则名称]
[一句话概括]

### N.1 [子条款]
...

---

## 治理
本宪法具有最高优先级，其效力高于任何 `CLAUDE.md` 或单次会话中的指令。
```

### Step 4：收尾

- 确认用户对全文满意
- 提示下一步：用 `/commit` 提交，然后开始编写 `CLAUDE.md`

## 注意事项

- 宪法是开发哲学，不是技术文档——不写具体的命令、工具、配置
- 每条原则保持简洁，避免冗长；能一句话说清的不用三句话
- 用中文编写，代码示例中的标识符用英文
